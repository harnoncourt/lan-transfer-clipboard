# 安全模型

## 当前安全状态

当前 MVP 默认信任同一局域网，不提供生产级安全保证。

当前没有实现：

- 身份认证。
- 设备配对。
- 传输加密。
- 请求签名。
- 接收确认。
- 文件大小限制。
- 恶意内容扫描。

因此当前版本只适合在可信网络中验证功能，不建议在公共 Wi-Fi 或不可信局域网中使用。

## 威胁模型

需要考虑的风险：

- 同一局域网内的陌生设备伪造 `hello`。
- 陌生设备向本机推送剪贴板文本。
- 陌生设备向本机上传大量文件占满磁盘。
- 中间人监听 HTTP 明文内容。
- 攻击者伪造文件名尝试路径穿越。
- 攻击者发送畸形 JSON 或超大请求体。

当前已处理：

- 文件名会替换常见非法路径字符。
- 收到自己的 `deviceId` 会忽略。
- 未识别 UDP packet 会忽略。

当前未充分处理：

- 请求来源校验。
- 文件大小限制。
- payload 加密。
- 内容完整性校验。
- 拒绝服务防护。

## 生产化安全目标

### P0：必须完成

- 首次使用设备配对。
- 未配对设备默认不可发送内容。
- 每次接收前展示来源和内容类型。
- 文件大小限制。
- 剪贴板文本长度限制。
- HTTP 请求超时。
- 接收目录空间检查。

### P1：强烈建议

- 每台设备生成长期密钥对。
- 使用配对码或 QR code 交换公钥。
- 请求签名，防止伪造发送方。
- payload 加密。
- 文件 SHA-256 校验。
- 接收历史中展示来源设备。

### P2：体验和治理

- 黑名单/信任设备管理。
- 自动删除过期接收记录。
- 危险文件类型提示。
- 传输审计日志。

## 推荐配对流程

1. A 发现 B。
2. A 请求配对。
3. B 展示 6 位配对码或 QR code。
4. A 输入配对码或扫描 QR code。
5. 双方交换公钥。
6. 双方保存对方设备 ID、公钥、设备名和首次信任时间。

## 推荐传输安全流程

1. 发送端生成随机会话密钥。
2. 使用接收端公钥加密会话密钥。
3. 使用会话密钥加密 payload。
4. 使用发送端私钥签名请求摘要。
5. 接收端验证签名和设备信任状态。
6. 接收端解密 payload。
7. 文件保存后校验 hash。

## 隐私说明建议

产品发布前应向用户明确说明：

- 数据只在局域网内传输。
- 不上传云端。
- 未配对设备不能主动推送内容。
- 用户可以删除信任设备。
- 用户可以清空接收历史。
